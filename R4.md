# R4: consume `iceberg-core` test JAR; arrest abstract-suite drift

Tracks the change that resolves the previously-listed errata T4 (the
abstract `CatalogTests` / `CatalogTransactionTests` suites were
duplicated between `iceberg/core/src/test/` and `fileio-catalog/src/test/`,
and had already drifted as of 2026-04-22). See also
[iceberg_refine.md#R4](../iceberg/iceberg_refine.md). The fileio-catalog
project no longer ships its own copies of the abstract suites; it consumes
them from the iceberg-core test JAR.

## What changed

### iceberg side

No build change required. The Iceberg fork already publishes a `tests`
classifier JAR for every non-BOM, non-openapi project (see
`iceberg/deploy.gradle:45–88`). After
`./gradlew publishToMavenLocal …`, the local Maven repository contains:

```
~/.m2/repository/org/apache/iceberg/iceberg-core/1.11.0-SNAPSHOT/iceberg-core-1.11.0-SNAPSHOT-tests.jar
~/.m2/repository/org/apache/iceberg/iceberg-api/1.11.0-SNAPSHOT/iceberg-api-1.11.0-SNAPSHOT-tests.jar
```

Both JARs include the abstract suites (`CatalogTests`,
`CatalogTransactionTests`) plus the test helpers they depend on
(`TestHelpers`, `TestBase`, `TableUtil`, etc.). The earlier R4 spec assumed
the test JAR had to be wired into the publication; that step had already
landed.

### fileio-catalog side

1. **Deleted duplicates** — removed
   `src/test/java/org/apache/iceberg/catalog/CatalogTests.java` and
   `CatalogTransactionTests.java`. As of 2026-04-22 these had drifted
   from the upstream copies (the local `CatalogTests` was 2796 lines vs
   3328 upstream — missing imports, assertions, and one new abstract
   method). All future upstream changes are inherited automatically.

2. **`pom.xml`** — already declared the test-jar dependencies on
   `iceberg-core` and `iceberg-api` (lines 105–118 of `pom.xml`). No
   change needed.

3. **Adapt concrete subclasses to the upstream API** — three local
   subclasses extend the abstract suites: `TestS3Catalog`,
   `GCSCatalogTest`, `ADLSCatalogTest`. They previously overrode
   fork-only hooks that no longer exist upstream; the migration is
   mechanical:

   | fork-only hook (deleted local copy)                        | upstream replacement                              |
   | ---------------------------------------------------------- | ------------------------------------------------- |
   | `protected String cannedTableLocation()`                   | `protected String baseTableLocation(TableIdentifier)` |
   | `protected boolean supportsConcurrentCreate() { return false; }` | (removed; upstream test runs unconditionally) |
   | (new abstract)                                             | `protected C initCatalog(String catalogName, Map<String,String> additionalProperties)` |

   Each subclass now:
   - Overrides `baseTableLocation(TableIdentifier)` to compose the cloud
     warehouse URL using the identifier's namespace + name (matching
     the upstream `BASE_TABLE_LOCATION` shape).
   - Adds an `initCatalog(...)` override that constructs a fresh
     `FileIOCatalog` with the additional properties merged in (used by
     `testCatalogWithCustomMetricsReporter`).
   - Drops the `supportsConcurrentCreate()` override. The upstream
     `testConcurrentCreateTransaction` now runs on the cloud subclasses
     when credentials are provided. If a real workload hits a provider
     where concurrent-create semantics differ, that's a separate
     conversation — gating it via a fork-only hook was hiding rather
     than solving the question.

   `TestS3FileIOCatalogTransaction`, `GCSFileIOCatalogTransactionTests`,
   and `ADLSFileIOCatalogTransactionTests` extend
   `CatalogTransactionTests`, which still requires only `catalog()` —
   no change needed.

4. **`CLAUDE.md`** — clarifies that `publishToMavenLocal` already emits
   the test JARs and fileio-catalog consumes them.

## How to verify

After publishing iceberg locally, fileio-catalog's unit test run is the
authoritative check:

```bash
cd iceberg && ./gradlew publishToMavenLocal -x test -x integrationTest -x generateGitProperties
cd ../fileio-catalog && mvn test
```

`Tests run: 208, Failures: 0, Errors: 0, Skipped: 0` is unchanged from
before R4, confirming that:
- the abstract suites resolved cleanly from the test-jar dependency,
- the concrete subclass migration compiles, and
- no in-project test regressed.

To inspect the consumed classpath:

```bash
mvn dependency:tree -Dscope=test 2>&1 | grep iceberg-core
# org.apache.iceberg:iceberg-core:jar:1.11.0-SNAPSHOT:compile
# org.apache.iceberg:iceberg-core:test-jar:tests:1.11.0-SNAPSHOT:test
```

## What this does NOT do

- Does not narrow the published test JAR. The JAR currently bundles
  every class in `iceberg-core/src/test/java`, not just
  `CatalogTests` and `CatalogTransactionTests`. That's fine in
  practice — the extra classes don't conflict with anything in
  fileio-catalog — but if a future upstream test pulls in a helper
  whose runtime dependencies are absent from the fileio-catalog
  classpath, the right fix is to add a selective `include` block in
  `iceberg/deploy.gradle`'s `testJar` task. The R4 spec proposes one;
  apply it then if needed.

- Does not address D2 (strict mixed-mode rejection) or T2/T3 (cloud
  inline-ML coverage and Testcontainers lifecycle). Those remain in
  errata.md.

## Why the spec was partially wrong

The original R4 spec at `iceberg/iceberg_refine.md` opens with "the test
JAR is not published by `./gradlew publishToMavenLocal`." That was not
the case at the time of writing — `iceberg/deploy.gradle:88` already had
`artifact testJar` in the publication block, and
`iceberg-core-1.11.0-SNAPSHOT-tests.jar` was already on disk locally.
The spec's step 1 (wire the test JAR into the publication) was therefore
unnecessary. Steps 2 and 3 (add the Maven dependency, delete the
duplicates) plus the subclass-migration work above are the actual fix.
